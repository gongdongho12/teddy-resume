# LINE Pay Plus Application Context - 2026-06-21

이 문서는 LINE Pay Plus 및 유사한 결제/리워드/커머스 백엔드 포지션 지원서를 다시 작성할 때 재사용하기 위한 컨텍스트이다.
아래 내용은 오늘 대화에서 정리한 사실, 표현 방향, 피해야 할 표현을 모은 것이다.

## 지원 대상 포지션 요약

- 회사/조직: LINE Pay Plus
- 도메인: LINE Pay 기반 결제 플랫폼, 리워드 광고, 커머스
- 주요 업무:
  - LINE Pay 백엔드 시스템 개발 및 API 구현
  - 광고/리워드의 대규모 트래픽 환경에서 동작하는 백엔드 설계, 개발, 운영
  - 프로토타이핑과 실험을 통한 서비스 개선
- 요구/우대 역량:
  - Java, Spring, Spring Boot, JPA
  - RDB 설계 및 쿼리
  - 대용량 트래픽 서비스 개발/운영
  - 테스트 코드
  - AI 기반 개발 도구 실무 활용
  - Git 협업 및 코드 리뷰
  - MSA, Kotlin, Spring Cloud, Redis/MongoDB/ElasticSearch, 모바일 API 설계, AI/데이터 연동, GitHub Actions

## 전체 포지셔닝

지원자는 단순히 맡은 백엔드 API만 구현하는 개발자가 아니라, 사용자가 실제로 기능을 완료하고 데이터 정합성이 맞는 상태에 도달할 때까지 앱, 프론트엔드, 백엔드, 인프라, 외부 연동 경계를 넘나들며 문제를 파고드는 개발자로 포지셔닝한다.

핵심 메시지는 다음과 같다.

- 결제/정산/리워드처럼 신뢰가 중요한 영역에서 상태 전이와 데이터 정합성을 중요하게 본다.
- 고객 입장에서는 앱 문제, 프론트엔드 문제, 백엔드 문제, 외부 PG 문제의 구분이 중요하지 않다. 기능이 동작하지 않는다는 사실만 남기 때문에, 내 포지션이 아니라고 선을 긋기보다 문제의 전체 흐름을 따라가야 한다고 생각한다.
- 결제 시스템에서는 단순히 PG API를 잘 호출하는 것보다, 실패/불일치/장애 이후 상태를 어떻게 분류하고 복구할지가 더 중요하다.
- 모바일 앱, 외부 계정/결제 연동, 백엔드 상태 관리, 운영 자동화, AI 개발 워크플로우까지 연결해서 문제를 해결한 경험을 강조한다.

## LG Uplus VoltUp 상세 업무 컨텍스트

전기차 충전 서비스의 결제/정산, 리워드/쿠폰, 사용자 앱 API, Flutter 기반 유저 앱, KakaoT 연동, 운영 자동화 영역을 담당했다.

강조할 내용:

- 단일 PG 중심 결제 구조를 KakaoPay, TossPayments, KakaoT 결제 등 멀티 벤더 구조로 확장했다.
- 벤더별 승인/취소/실패 처리 차이를 추상화하여 결제 도메인의 변경 범위를 줄였다.
- 충전 완료 후 결제 실패가 발생하는 미수 케이스를 안정적으로 관리하기 위해 트랜잭션 아웃박스 패턴에 기반한 이벤트 저장/발행 흐름을 설계했다.
- GCP Pub/Sub DLQ, 재시도 배치, Failover 상태 전이를 구성해 실패 이벤트 누락과 재시도 정체를 줄였다.
- 쿠폰, 포인트, U+ VIP 혜택 등 리워드성 정책을 결제 흐름과 함께 다뤘다.
- Hold -> Confirm -> Release 형태의 상태 전이를 통해 혜택 적용과 결제 정합성이 어긋나지 않도록 개선했다.
- Flutter 기반 유저 앱 기능 개발에도 참여했다.
- 결제수단 등록, KakaoT 연동 복귀, 차량 조회/등록, 충전 관련 사용자 흐름이 백엔드 API와 자연스럽게 이어지도록 개선했다.
- KakaoT 계정/결제수단 연동에서 외부 콜백, 앱 딥링크 복귀, 결제수단 활성화 API, partnerUserToken 검증 흐름을 설계했다.
- 계정 소유자와 결제수단 소유자가 불일치하는 문제를 방지했다.
- Voltbot의 에이전트 전환 및 컨텍스트 공유 기능을 직접 개발했다.
- 코드 정책 에이전트 및 AI 기반 코드/보안 리뷰 워크플로우를 여러 레포지토리에 실제 적용했다.
- 앞으로 또는 현재 확장 방향으로, 로그를 분석해 고객이 겪은 동작 흐름을 상세 기록으로 뽑는 로그 에이전트도 언급할 수 있다. 단, 완료된 기능처럼 과장하지 말고 "확장하려고 한다", "시도하고 있다" 정도로 표현한다.

## Kakao Style 상세 업무 컨텍스트

지그재그 커머스 플랫폼에서 상품, 가격, 멤버십, 외부 쇼핑 채널 연동 영역의 백엔드 개발을 담당했다.

강조할 내용:

- 상품 상세, 장바구니, 주문 등 여러 도메인에 흩어져 있던 가격 계산 로직을 통합 최종가 API로 정리했다.
- 쿠폰/프로모션/할인 정책이 일관되게 적용될 수 있도록 개선했다.
- 상품 매칭 구조를 기존 1:1 중심에서 N:M 구조로 확장했다.
- 동일 상품이 여러 쇼핑몰, 옵션, 외부 채널과 연결될 수 있는 데이터 모델과 API를 개선했다.
- Kafka, Athena 기반 데이터 흐름을 활용해 상품 매칭 및 가격 데이터 처리 구조를 고도화했다.
- 네이버 쇼핑, 구글 쇼핑 등 외부 채널에 제공하는 EP 데이터와 최저가 갱신 파이프라인을 개발/운영했다.
- 품절/가격 변경 등 부분 업데이트가 필요한 케이스를 처리했다.
- 레거시 멤버십/마일리지 시스템을 Spring Boot 기반 구조로 전환했다.
- 등급 산정 및 캐시 처리 흐름을 개선하여 서비스 중단 없이 안정적으로 마이그레이션했다.

## LINE Pay Plus 자기소개서 5문항 방향

### 1. 자신을 자유롭게 소개

관점:

- "정합성과 사용자 흐름을 함께 보는 개발자"로 소개한다.
- 결제/커머스/리워드에서 금액, 상태, 혜택이 조금만 어긋나도 신뢰가 깨진다는 점을 강조한다.
- 역할 경계보다 문제 해결을 우선하는 태도를 말한다.

넣을 수 있는 표현:

- 고객에게는 앱 문제인지, 백엔드 문제인지, 외부 연동 문제인지가 중요하지 않다.
- 기능이 끝까지 동작하고 데이터가 맞는 상태가 되어야 서비스가 완성된다고 생각한다.

### 2. LINE에서 왜 자신을 채용해야 하는지

관점:

- LINE Pay Plus의 결제/리워드/커머스 백엔드와 직접 연결되는 경험을 강조한다.
- 멀티 벤더 결제, 미수 복구, 리워드/쿠폰/포인트 정책, 모바일 앱 연동, AI 개발 워크플로우 경험을 묶는다.
- "공고에서 강조한 AI 기반 개발 방식"이라는 표현은 가능하지만, "LINE의 AI-driven 개발 문화를 잘 알고 있다"처럼 내부 문화를 아는 듯한 표현은 피한다.

### 3. 대표 프로젝트 세 가지

추천 프로젝트:

1. VoltUp 멀티 벤더 결제 및 미수/정산 안정화
2. KakaoT 계정/결제수단 연동 및 Flutter 앱 복귀 흐름
3. Kakao Style 최종가 API/상품 매칭/외부 쇼핑 채널 연동

보조 후보:

- VoltUp 리워드/쿠폰/포인트/U+ VIP 혜택 정책
- Voltbot 에이전트 전환/컨텍스트 공유 및 AI 리뷰 워크플로우

### 4. 기억에 남는 트러블슈팅

권장 주제:

- 결제 실패 이벤트가 누락되거나 재시도 흐름에서 정체될 수 있던 문제를 해결한 경험
- 충전 완료 후 결제 실패는 단순 주문 실패가 아니라 미수/정산 누락/사용자 제한으로 이어질 수 있음을 설명한다.
- 트랜잭션 아웃박스 패턴, DLQ, 재시도 배치, Failover 상태 전이를 넣어 기술적 깊이를 만든다.

피할 주제:

- "대표 카드 재시도", "기본적인 결제수단 실패 처리" 정도로 보이는 단순 이슈만 단독으로 쓰지 않는다.

### 5. AI 활용 생산성 향상

관점:

- 단순히 ChatGPT로 코드 작성했다가 아니라, 개발 워크플로우에 AI를 넣어 반복 작업을 줄이고 리뷰 품질을 높인 경험으로 작성한다.
- Voltbot은 전체를 본인이 만들었다고 쓰면 안 된다.
- 정확한 기여는 "에이전트 전환 기능과 컨텍스트 공유 기능을 직접 구현했다"이다.
- voltup-workflow의 코드 리뷰/보안 리뷰 로직은 여러 레포에서 실제 사용까지 적용했다.
- Code Policy Agent와 함께 사용하여 팀의 코드 정책을 리뷰 맥락에 녹이는 방향을 강조한다.
- Log Agent는 로그를 분석해 고객이 겪은 동작 흐름을 상세 기록으로 뽑는 방향으로 진행/확장 중이라고 표현한다.

## 선을 넘지 말아야 할 표현

- Voltbot 대부분을 직접 만들었다고 쓰지 않는다.
- LINE의 내부 AI-driven 개발 문화를 이미 알고 있는 것처럼 말하지 않는다.
- 정량 지표가 확인되지 않은 경우 "몇 퍼센트 개선", "몇 배 향상" 같은 수치는 쓰지 않는다.
- "대규모 트래픽에 능숙"처럼 추상적으로만 말하지 말고, 결제/리워드/커머스의 구체 흐름으로 보여준다.
- KakaoT 연동은 단순 외부 API 연동이 아니라, 계정/결제수단/앱 복귀/활성화/소유자 검증까지 연결된 사용자 흐름으로 설명한다.
- AI 활용은 과장하지 않고 실제 구현한 자동화, 리뷰, 컨텍스트 공유, 로그 분석 계획을 기준으로 쓴다.

## 제출용 문체 가이드

- 톤: 단정하고 구체적으로 쓴다.
- 문장: "무엇을 했다"보다 "어떤 문제를 보고, 어떤 기준으로 설계했고, 그 결과 어떤 안정성을 확보했다"로 쓴다.
- 기술명은 문제 해결 맥락 안에서만 사용한다.
- 자기소개서에서는 같은 이야기를 반복하지 않도록 각 문항의 초점을 나눈다.
  - 자기소개: 개발 철학
  - 채용 이유: 포지션 적합성
  - 프로젝트: 기술 구현 상세
  - 트러블슈팅: 장애/복구 설계
  - AI: 개발 생산성/품질 자동화
